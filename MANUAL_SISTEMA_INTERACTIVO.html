🚀 Sistema de Gestión de Inventario

Manual Interactivo Completo - Sistema de Reposición Inteligente

📊 Vista General del Sistema

🎯 Objetivo del Sistema

Sistema automatizado de gestión de inventario que calcula cuánto stock mantener y cuándo reordenar, basado en datos históricos de ventas, tiempos de entrega y niveles de servicio deseados.

🔑 Conceptos Clave

📊 Venta Diaria

Promedio de unidades vendidas por día en un período determinado. Base para todos los cálculos.

🎯 Stock Objetivo

Cantidad ideal de inventario para cubrir ventas durante el tiempo de reposición + stock de seguridad.

📦 Cantidad Sugerida

Unidades que debes comprar para alcanzar el stock objetivo, considerando stock actual y en tránsito.

🚢 Stock en Tránsito

Productos ya ordenados pero no recibidos. Reduce la cantidad a comprar.

⏱️ Lead Time

Tiempo total desde que ordenas hasta que recibes: Fabricación + Envío + Buffer

💰 Rentabilidad

Cálculo de margen considerando costo FOB, envío, comisiones ML y costos operacionales.

🏗️ Arquitectura del Sistema

1 Carga de Datos - Excel/CSV → Supabase (productos, ventas, compras, packs)
2 Procesamiento - Descomposición de packs, cálculo de venta diaria
3 Análisis - Stock objetivo, cantidad sugerida, estados de reposición
4 Exportación - Filtros por estado, priorización, agrupación por proveedor
5 Purchase Orders - Generación de órdenes, tracking de recepción
6 Containers - Optimización de espacio, cálculo de CBM

⚡ Características Principales

Característica Descripción Beneficio
Cálculo Automático Calcula venta diaria, stock objetivo y cantidad sugerida automáticamente Elimina cálculos manuales propensos a errores
Sistema de Packs Descompone ventas de packs en componentes individuales Cálculos precisos de reposición por producto
Priorización Clasifica productos por urgencia (CRITICAL, PARTIAL, etc.) Enfoca recursos en productos críticos
Stock en Tránsito Considera compras confirmadas pero no recibidas Evita sobre-ordenar productos ya en camino
Rentabilidad Calcula margen, ROI y costos totales Decisiones de compra basadas en rentabilidad

📥 Flujo Completo de Datos

1️⃣ Carga de Productos

📦 Estructura de Datos de Productos ▼

Campos requeridos:

  • sku: Identificador único (ej: "PROD-001")
  • descripcion: Nombre del producto
  • cbm: Volumen en metros cúbicos (ej: 0.05)
  • costo_fob_rmb: Costo FOB en yuanes (ej: 50)
  • stock_actual: Inventario actual (ej: 150)
  • precio_venta_ml: Precio en MercadoLibre (ej: 15000)
Ejemplo de Producto:
{ "sku": "LAMP-001", "descripcion": "Lámpara LED 10W", "cbm": 0.08, "costo_fob_rmb": 45, "stock_actual": 200, "precio_venta_ml": 12990, "proveedor": "Shenzhen Lighting Co." }
⚠️ Validaciones:
  • SKU debe ser único (no duplicados)
  • CBM debe ser > 0
  • Costo FOB debe ser > 0
  • Si el producto no existe, las ventas/compras serán rechazadas
🔄 Proceso de Carga ▼
Paso 1: Validación
Se verifica que el SKU no esté duplicado, que CBM y costo sean válidos
Paso 2: Normalización
SKU se convierte a mayúsculas, espacios múltiples se eliminan
Paso 3: Insert Batch
Se insertan en bloques de 100 productos para mejor performance
Paso 4: Confirmación
Sistema reporta cuántos productos fueron creados exitosamente

2️⃣ Carga de Ventas

📊 Estructura de Datos de Ventas ▼

Campos requeridos:

  • sku: SKU del producto (debe existir en productos)
  • cantidad: Unidades vendidas (> 0)
  • fecha_venta: Fecha de la venta (formato: YYYY-MM-DD)
Ejemplo de Venta:
{ "sku": "LAMP-001", "cantidad": 5, "fecha_venta": "2025-01-15" }
✅ Validaciones Automáticas:
  • El producto con ese SKU debe existir (foreign key)
  • Cantidad debe ser > 0
  • Fecha debe ser válida
  • Se detectan duplicados (mismo SKU + misma fecha)
🎯 Proceso de Carga de Ventas ▼
1 Verificación de Productos - Se valida que todos los SKUs existen
2 Detección de Duplicados - Se buscan ventas con mismo SKU + fecha
3 Descomposición de Packs - Si el SKU es un pack, se descompone en componentes
4 Insert Batch - Se insertan en bloques de 500 registros
5 Actualización de Vista - Se refresca la vista materializada de venta diaria

3️⃣ Carga de Compras

🛒 Estructura de Datos de Compras ▼

Campos requeridos:

  • sku: SKU del producto
  • cantidad: Unidades compradas
  • fecha_compra: Fecha de la orden
  • status_compra: Estado de la orden

Estados válidos de compra:

Estado Descripción Cuenta como Stock en Tránsito
cotizacion Cotización solicitada ❌ No
cotizacion_confirmada Cotización aprobada ❌ No
confirmado Orden confirmada con proveedor ✅ Sí
fabricacion En proceso de fabricación ✅ Sí
fabricacion_completa Fabricación completada ✅ Sí
en_transito En camino a bodega ✅ Sí
llegado Recibido en bodega ❌ No (ya suma al stock actual)

4️⃣ Carga de Packs

🔥 Importante: Los packs son productos compuestos. Cuando vendes 1 pack, en realidad estás vendiendo múltiples productos individuales.
Ejemplo de Pack:

Pack "SET-LIVING-01" contiene:

  • 2x "LAMP-001" (Lámpara LED)
  • 1x "TABLE-001" (Mesa auxiliar)
  • 4x "CUSHION-001" (Cojín decorativo)

Si vendes 3 unidades del pack, el sistema registra:

  • 6x LAMP-001 (3 packs × 2 unidades)
  • 3x TABLE-001 (3 packs × 1 unidad)
  • 12x CUSHION-001 (3 packs × 4 unidades)

🔢 Cálculos del Sistema

1️⃣ Venta Diaria

Venta Diaria = Total Vendido / Días del Período
Ejemplo:

Producto: LAMP-001

  • Total vendido en 90 días: 450 unidades
  • Venta Diaria = 450 / 90 = 5 unidades/día

2️⃣ Stock Objetivo

Stock Objetivo = Venta Diaria × (Lead Time + Stock Saludable)
Ejemplo:

Producto: LAMP-001

  • Venta Diaria: 5 unidades/día
  • Lead Time: 90 días (30 fabricación + 60 envío)
  • Stock Saludable: 60 días
  • Stock Objetivo = 5 × (90 + 60) = 5 × 150 = 750 unidades
💡 Explicación:

Necesitas suficiente stock para cubrir:

  • Lead Time (90 días): Lo que venderás mientras esperas la próxima orden
  • Stock Saludable (60 días): Buffer de seguridad para imprevistos

3️⃣ Cantidad Sugerida

⚠️ Fórmula Corregida: La cantidad sugerida NO es simplemente Stock Objetivo - Stock Actual - Stock en Tránsito.
Debes considerar el consumo durante el lead time.
Paso 1: Consumo Durante Lead Time = Venta Diaria × Lead Time
Paso 2: Stock Proyectado a la Llegada = Stock Actual + Stock en Tránsito - Consumo Durante Lead Time
Paso 3:
  Si Stock Proyectado < 0: Cantidad Sugerida = Stock Objetivo
  Si Stock Proyectado ≥ 0: Cantidad Sugerida = MAX(0, Stock Objetivo - Stock Proyectado)
Ejemplo:

Producto: LAMP-001

  • Venta Diaria: 5 unidades/día
  • Lead Time: 90 días
  • Stock Saludable: 60 días
  • Stock Actual: 200 unidades
  • Stock en Tránsito: 150 unidades

Cálculo paso a paso:

  1. Stock Objetivo = 5 × 60 = 300 unidades
  2. Consumo Durante Lead Time = 5 × 90 = 450 unidades
  3. Stock Proyectado = 200 + 150 - 450 = -100 unidades (negativo!)
  4. Como Stock Proyectado < 0: Cantidad Sugerida = 300 unidades (Stock Objetivo completo)
Ejemplo 2 (Stock Proyectado Positivo):

Producto: TABLE-001

  • Venta Diaria: 1 unidad/día
  • Lead Time: 90 días
  • Stock Saludable: 60 días
  • Stock Actual: 100 unidades
  • Stock en Tránsito: 50 unidades

Cálculo paso a paso:

  1. Stock Objetivo = 1 × 60 = 60 unidades
  2. Consumo Durante Lead Time = 1 × 90 = 90 unidades
  3. Stock Proyectado = 100 + 50 - 90 = 60 unidades (positivo)
  4. Cantidad Sugerida = MAX(0, 60 - 60) = 0 unidades

4️⃣ Días de Inventario

Días de Inventario = Stock Actual / Venta Diaria
Ejemplo:

Producto: LAMP-001

  • Stock Actual: 200 unidades
  • Venta Diaria: 5 unidades/día
  • Días de Inventario = 200 / 5 = 40 días
⚠️ Caso Especial: Si Venta Diaria = 0, Días de Inventario = ∞ (Infinito)

5️⃣ Estados de Reposición

Estado Condición Acción Recomendada
CRITICAL Cantidad Sugerida > 0 Y Stock en Tránsito = 0 🚨 Ordenar URGENTE
PARTIAL Cantidad Sugerida > 0 Y Stock en Tránsito > 0 ⚠️ Considerar orden adicional
COVERED Cantidad Sugerida = 0 Y Stock en Tránsito > 0 ✅ Cubierto por orden en tránsito
SUFFICIENT_STOCK Cantidad Sugerida = 0 Y Stock en Tránsito = 0 👍 Stock suficiente
OVER_ORDERED Stock Actual + Stock en Tránsito > Stock Objetivo × 1.5 ⚠️ Sobre stock - no ordenar

6️⃣ Rentabilidad

Cálculo Paso a Paso:

1 Costo FOB en CLP = Costo FOB RMB × Tasa Cambio RMB
2 Envío por Unidad = Envío Total Container / Unidades en Container
3 Costo Total = Costo FOB CLP + Envío por Unidad
4 Comisión ML = Precio Venta × 16%
5 Envío ML = $5,000 (o $0 si precio > $60,000)
6 Costo Operacional = Precio Venta × 15%
7 Margen Bruto = Precio Venta - Costo Total - Comisión - Envío - Costo Op
8 ROI % = (Margen / Costo Total) × 100
Ejemplo Completo de Rentabilidad:

Producto: LAMP-001

  • Costo FOB: 45 RMB × 130 (tasa) = 5,850 CLP
  • Envío: 500,000 CLP container / 1,000 unidades = 500 CLP
  • Costo Total = 5,850 + 500 = 6,350 CLP
  • Precio Venta ML: 12,990 CLP
  • Comisión ML: 12,990 × 16% = 2,078 CLP
  • Envío ML: 5,000 CLP
  • Costo Operacional: 12,990 × 15% = 1,949 CLP
  • Margen = 12,990 - 6,350 - 2,078 - 5,000 - 1,949 = -2,387 CLP ⚠️ NO RENTABLE

📦 Sistema de Packs

¿Qué es un Pack?

Un pack es un producto compuesto por múltiples productos individuales. Cuando vendes un pack, en realidad estás vendiendo varios productos a la vez.

🔄 Descomposición de Ventas de Packs

Escenario Completo:

Definición del Pack:

Pack: SET-LIVING-01 Componentes: - 2x LAMP-001 (Lámpara LED) - 1x TABLE-001 (Mesa auxiliar) - 4x CUSHION-001 (Cojín)

Ventas del mes:

  • 3x SET-LIVING-01 (pack completo)
  • 5x LAMP-001 (vendida individualmente)
  • 2x TABLE-001 (vendida individualmente)

Descomposición automática:

1 3x SET-LIVING-01 → Se descompone en:
  • 6x LAMP-001 (3 packs × 2 lámparas)
  • 3x TABLE-001 (3 packs × 1 mesa)
  • 12x CUSHION-001 (3 packs × 4 cojines)
2 Se consolidan con ventas individuales:
  • LAMP-001: 6 (del pack) + 5 (individual) = 11 unidades
  • TABLE-001: 3 (del pack) + 2 (individual) = 5 unidades
  • CUSHION-001: 12 (del pack) + 0 (individual) = 12 unidades
3 Se calcula venta diaria de cada componente:
  • LAMP-001: 11 / 30 días = 0.37 unidades/día
  • TABLE-001: 5 / 30 días = 0.17 unidades/día
  • CUSHION-001: 12 / 30 días = 0.40 unidades/día
⚠️ Error Común: Si NO descompones los packs:
  • LAMP-001 mostraría solo 5 ventas → Venta diaria incorrecta
  • Sistema sugeriría comprar menos lámparas de las necesarias
  • Resultado: Quiebre de stock cuando vendan más packs

✅ Validaciones del Sistema de Packs

  • ✅ Todos los componentes del pack deben existir como productos
  • ✅ Las cantidades de componentes deben ser > 0

  • ✅ La descomposición es automática al cargar ventas
  • ✅ El cálculo de venta diaria incluye automáticamente las ventas descompuestas

📈 Análisis y Reportes

🎯 Vista Materializada: sku_venta_diaria_mv

¿Qué es?

Una tabla pre-calculada que almacena la venta diaria de cada producto, incluyendo ventas descompuestas de packs. Se actualiza automáticamente cuando cargas nuevas ventas.

Estructura de la Vista:
SELECT sku, venta_diaria, total_vendido, dias_periodo, fecha_inicio, fecha_fin FROM sku_venta_diaria_mv

📊 Análisis Principal

Proceso de Análisis:

1 Obtener Venta Diaria - Desde vista materializada
2 Calcular Stock Objetivo - Venta Diaria × (Lead Time + Buffer)
3 Obtener Stock en Tránsito - Sumar compras confirmadas/en_transito/fabricacion
4 Calcular Cantidad Sugerida - Stock Objetivo - Stock Actual - Stock en Tránsito
5 Determinar Estado - CRITICAL, PARTIAL, COVERED, etc.
6 Calcular Rentabilidad - Margen, ROI, costos totales

📑 Ordenamiento de Productos en Exports

🎯 Criterio Principal: VALOR TOTAL

Los productos se ordenan de MAYOR a MENOR valor total, priorizando aquellos que requieren más inversión.

Valor Total = Cantidad Sugerida × Precio Venta ML
Ejemplo de Ordenamiento:

Tienes 3 productos que necesitan reposición:

Posición SKU Cantidad Sugerida Precio Unitario Valor Total Prioridad
1º SOFA-2P-01 20 unidades $89,990 $1,799,800 CRÍTICA
2º TABLE-AUX-01 85 unidades $29,990 $2,549,150 CRÍTICA
3º LAMP-LED-10W 400 unidades $12,990 $5,196,000 CRÍTICA

⚠️ Ordenamiento CORRECTO (por Valor Total):

  1. 1º LAMP-LED-10W: $5,196,000 (mayor inversión)
  2. 2º TABLE-AUX-01: $2,549,150
  3. 3º SOFA-2P-01: $1,799,800 (menor inversión)
💡 Razón: Priorizar por valor total te ayuda a tomar decisiones de inversión informadas. Si tienes presupuesto limitado, verás primero los productos que requieren mayor capital.

🏷️ Etiquetas de Prioridad (Visuales)

Nota: Las etiquetas de prioridad (CRÍTICA, ALTA, MEDIA, BAJA) son solo visuales para ayudarte a identificar rápidamente el impacto financiero.
Etiqueta Criterio Significado
CRÍTICA Valor Total > $500,000 CLP Requiere alta inversión - revisar cuidadosamente
ALTA Valor Total > $200,000 CLP Inversión moderada - importante

🛒 Proceso de Compra

📋 Flujo Completo de Purchase Orders

1. Análisis y Exportación
Sistema genera lista de productos que necesitan reposición, filtrada por prioridad
2. Agrupación por Proveedor
Productos se agrupan por proveedor para enviar órdenes consolidadas
3. Generación de Order Number
Formato: ORD-YYYYMMDD-XXXXX (ej: ORD-20250116-12345)
4. Creación de Purchase Order
Se crea registro con estado "PENDING"
5. Envío al Proveedor
Se envía orden con cantidades, especificaciones y fecha de entrega requerida
6. Confirmación
Proveedor confirma → Estado cambia a "CONFIRMED" → Se crea registro en "compras"
7. Tracking
Se actualiza estado de compra: fabricacion → en_transito → llegado
8. Recepción
Al llegar, se actualiza stock_actual y estado de compra a "llegado"

📊 Estados de Purchase Order

Estado Descripción Siguiente Paso
PENDING Orden creada, esperando envío al proveedor Enviar al proveedor
SENT Orden enviada al proveedor, esperando confirmación Esperar confirmación
CONFIRMED Proveedor confirmó la orden Crear registro en "compras"
PARTIAL Recibido parcialmente Esperar resto del pedido
RECEIVED Completamente recibido Actualizar stock_actual
CANCELLED Orden cancelada Crear nueva orden si es necesario

🚢 Containers y Optimización

Cálculo de CBM de Container:

Productos a ordenar:

  • 400x LAMP-001 (CBM: 0.08) = 32 CBM
  • 200x TABLE-001 (CBM: 0.15) = 30 CBM
  • 500x CUSHION-001 (CBM: 0.02) = 10 CBM

Total CBM: 72 m³

Container de 20': 30 m³ → Se necesitan 3 containers
Container de 40': 60 m³ → Se necesitan 2 containers
Container de 40' HC: 68 m³ → Se necesitan 2 containers
Recomendación: Usar 1x 40'HC (68m³) + 1x 20' (30m³) = 98m³ total
Utilización: 72/98 = 73.5% - Eficiente

💡 Validaciones al Crear Purchase Order

  • ✅ Todos los productos deben existir
  • ✅ Cantidades deben ser > 0
  • ✅ CBM total no debe exceder capacidad del container
  • ✅ Order number debe ser único
  • ✅ Proveedor debe estar registrado
⚠️ Alerta de Sobre Capacidad:

Si CBM total excede capacidad del container, sistema alertará y sugerirá:

  • Agregar otro container
  • Reducir cantidades de productos de baja prioridad
  • Mover algunos productos a próxima orden

💾 Estructura de Base de Datos

📊 Tablas Principales

🏗️ products ▼
CREATE TABLE products ( id SERIAL PRIMARY KEY, sku VARCHAR(255) UNIQUE NOT NULL, descripcion TEXT NOT NULL, cbm DECIMAL(10,4), costo_fob_rmb DECIMAL(10,2), stock_actual INTEGER DEFAULT 0, precio_venta_ml INTEGER, proveedor VARCHAR(255), desconsiderado BOOLEAN DEFAULT FALSE, created_at TIMESTAMP DEFAULT NOW() );

Índices importantes:

  • UNIQUE INDEX idx_products_sku ON products(sku)
  • INDEX idx_products_proveedor ON products(proveedor)
📊 ventas ▼
CREATE TABLE ventas ( id SERIAL PRIMARY KEY, sku VARCHAR(255) NOT NULL, cantidad INTEGER NOT NULL CHECK (cantidad > 0), fecha_venta TIMESTAMP NOT NULL, created_at TIMESTAMP DEFAULT NOW(), FOREIGN KEY (sku) REFERENCES products(sku) );

Índices importantes:

  • INDEX idx_ventas_sku ON ventas(sku)
  • INDEX idx_ventas_fecha ON ventas(fecha_venta)
  • INDEX idx_ventas_sku_fecha ON ventas(sku, fecha_venta)
⚠️ Duplicados: Se recomienda agregar constraint UNIQUE(sku, fecha_venta) para prevenir duplicados.
🛒 compras ▼
CREATE TABLE compras ( id SERIAL PRIMARY KEY, sku VARCHAR(255) NOT NULL, cantidad INTEGER NOT NULL CHECK (cantidad > 0), fecha_compra DATE NOT NULL, status_compra VARCHAR(50) NOT NULL, purchase_order_id INTEGER, created_at TIMESTAMP DEFAULT NOW(), FOREIGN KEY (sku) REFERENCES products(sku), CHECK (status_compra IN ( 'cotizacion', 'cotizacion_confirmada', 'confirmado', 'fabricacion', 'fabricacion_completa', 'en_transito', 'llegado' )) );
📦 packs ▼
CREATE TABLE packs ( id SERIAL PRIMARY KEY, pack_sku VARCHAR(255) NOT NULL, producto_sku VARCHAR(255) NOT NULL, cantidad INTEGER NOT NULL CHECK (cantidad > 0), created_at TIMESTAMP DEFAULT NOW(), FOREIGN KEY (pack_sku) REFERENCES products(sku), FOREIGN KEY (producto_sku) REFERENCES products(sku), UNIQUE (pack_sku, producto_sku) );
✅ Constraint UNIQUE: Previene duplicados de la misma combinación pack + producto
🚢 containers ▼
CREATE TABLE containers ( id SERIAL PRIMARY KEY, container_number VARCHAR(50) UNIQUE NOT NULL, capacidad_cbm DECIMAL(10,2) NOT NULL, status VARCHAR(50) NOT NULL, fecha_envio DATE, fecha_llegada_estimada DATE, created_at TIMESTAMP DEFAULT NOW(), CHECK (status IN ('CREATED', 'IN_TRANSIT', 'DELIVERED')) );
📋 purchase_orders ▼
CREATE TABLE purchase_orders ( id SERIAL PRIMARY KEY, order_number VARCHAR(50) UNIQUE NOT NULL, proveedor VARCHAR(255) NOT NULL, status VARCHAR(50) NOT NULL, fecha_orden DATE NOT NULL, fecha_entrega_estimada DATE, total_items INTEGER, total_cbm DECIMAL(10,2), created_at TIMESTAMP DEFAULT NOW(), CHECK (status IN ( 'PENDING', 'SENT', 'CONFIRMED', 'PARTIAL', 'RECEIVED', 'CANCELLED' )) );

🔍 Vista Materializada: sku_venta_diaria_mv

CREATE MATERIALIZED VIEW sku_venta_diaria_mv AS SELECT sku, SUM(cantidad) as total_vendido, COUNT(DISTINCT DATE(fecha_venta)) as dias_periodo, SUM(cantidad)::DECIMAL / GREATEST(COUNT(DISTINCT DATE(fecha_venta)), 1) as venta_diaria, MIN(fecha_venta) as fecha_inicio, MAX(fecha_venta) as fecha_fin FROM ventas GROUP BY sku; -- Índice para búsquedas rápidas CREATE UNIQUE INDEX idx_sku_venta_diaria_mv_sku ON sku_venta_diaria_mv(sku);
💡 Ventaja: Cálculo de venta diaria pre-calculado → Análisis 100x más rápido

🔗 Relaciones entre Tablas

products ← (FK) ventas.sku
products ← (FK) compras.sku
products ← (FK) packs.pack_sku
products ← (FK) packs.producto_sku
purchase_orders ← (FK) compras.purchase_order_id
✅ Integridad Referencial:
  • No se pueden crear ventas de productos inexistentes
  • No se pueden crear compras de productos inexistentes
  • No se pueden crear packs con componentes inexistentes
  • No se puede eliminar un producto con ventas/compras asociadas
